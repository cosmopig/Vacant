## 任務範圍

Vacant 的設計假設是「token 免費的未來」，但 demo 必須在「token 還貴的現在」做出來給審查者看。你要設計**可在畢業專題範圍內完成的 MVP 與 demo 計畫**，並設計**評估指標**。

## 你要設計

1. **MVP 範圍界定**（最小可展示的網路活動）
   - 至少要有的元件：Registry、Vacant Runtime（≥3 個個體）、客戶端 SDK、Aggregator
   - 至少要展示的行為：peer review、spawn、sink、composition link、idle-time 演化（壓縮版）
   - 至少要拿到的數據：一個 reputation 隨時間變化的圖、一次「失敗→競爭者誕生」的完整 trace

2. **「Token 免費」的 demo 模擬**
   - 用什麼模型：Qwen 2.5 7B / Llama 3.2 / Gemma 等本地小模型
   - 用什麼 inference：Ollama / llama.cpp / vLLM
   - 模型多樣性：刻意配 3-5 種不同 base model 展示跨模型多元性
   - 預算：估算硬體（單機 RTX 4090 / Mac M-series 32GB+）

3. **Time compression**
   - 真實 adoption signal 要 7 天 → demo 壓縮成 7 分鐘
   - heartbeat 間隔縮短（從 30 分鐘 → 30 秒）
   - 但要說明哪些壓縮是合理的、哪些可能造成失真

4. **Demo 場景**（劇本）
   - **場景 A**：法律問答 vacant + 醫療問答 vacant，使用者問跨領域問題 → 兩者組產業鏈
   - **場景 B**：故意放一個「會偷工減料」的 adversary vacant 進網路 → 展示 reputation 自然下降 → spawn 競爭者
   - **場景 C**：複合 vacant（行銷 vacant 自生美編子代）展示內部運作
   - **場景 D**：嘗試 Sybil 攻擊（同一開發者開 10 個假 vacant 互相 inflate）→ 展示為何無法奏效（與 P2 對齊）

5. **評估指標**
   - **任務正確率**：相比單 agent 是否提升（借用 GSM8K / MMLU / HumanEval）
   - **Reputation 收斂速度**：N 次互動後高/低 vacant 是否能被正確分類（借用 DRF Fig. 3）
   - **抗對抗測試**：放 K 個 adversary 後系統穩健性（借用 CrS adversary-majority test）
   - **多元性指數**：呼叫分布的 Gini / Shannon entropy
   - **誠實度**：self-eval 與 peer-eval 的相關係數
   - **失敗→競爭**：從第一次失敗到 spawn 替代者的時間

6. **失敗條件 / 不展示什麼**
   - **不展示**：MINJA 端到端攻擊（demo 範圍外）
   - **不展示**：完整聯邦 Registry（MVP 是單 Registry）
   - **誠實列出**：哪些 Vacant 主張在 MVP 規模下無法驗證（例如「網路自然產生跨模型多元性」需要更大網路）

7. **timeline 估計**
   - 如果一個人做：每個元件 N 天、整合 N 天、demo 場景 N 天、評估 N 天
   - 如果三人團隊：分工建議

8. **論文範圍建議**
   - 以這個 MVP + 評估數據能寫出什麼樣的論文
   - 哪些章節能寫滿、哪些是 future work
   - 引用三篇文獻（CrS / DRF / A-Trust）的位置
