# P6: A2A / MCP 整合介面

> 你是 P6-protocol pane (label: P6-protocol)。先讀 `/Users/cosmopig/Downloads/專題/architecture/BRIEFING.md`，再讀 `責任有效性分析` §2.1（MCP 與 A2A 並未處理「責任」）。

## 任務範圍

Vacant **不重做 A2A/MCP**，只在其上做「責任元層」。你要設計**精確的 envelope schema** 與 **整合介面**。

## 必查資源

請使用 WebFetch 或 WebSearch 取得最新文件：
- A2A spec：https://google.github.io/A2A/ 或 Linux Foundation 接管後的最新 spec
- MCP spec：https://modelcontextprotocol.io/
- A2A Agent Card schema 結構
- MCP server adapter 設計範式

## 你要設計

1. **Vacant Capability Card**（A2A Agent Card 的擴充）
   - 標準 A2A Agent Card 欄位（name, description, capabilities, endpoints, auth）
   - + Vacant 加上：`vacant_id` (Ed25519 公鑰), `parent_id`, `base_model`, `version`, `attestations[]`, `stake`, `reputation_snapshot_url`, `sink_status`
   - 給完整 JSON 範例（at least 3 個：simple vacant、composite vacant、newly-spawned vacant）

2. **Vacant Envelope**（A2A message 的擴充）
   - 標準 A2A message：method, params, id
   - + Vacant 加上：`caller_vacant_id`, `caller_signature`, `idempotency_key`, `expected_dim_weights{}`, `chain_attestation[]`（呼叫鏈上每個 vacant 的簽章）
   - 給完整 JSON 範例

3. **Review Envelope**（評分回送 Registry）
   - `call_id`, `reviewer_vacant_id`, `reviewee_vacant_id`, `dim_scores{factual, logical, relevance, honesty, adoption}`, `confidence`, `signature`, `evidence_pointer`
   - peer review（idle-time，不關聯 specific call）的變體

4. **MCP Server Adapter**（讓任何 MCP client 能用 vacant）
   - 一個 MCP server，向上暴露 `vacant_query` / `vacant_call` / `vacant_review` 三個 tool
   - 內部走 A2A 到實際 vacant
   - 給 MCP server.json 設定範例 + tool schema

5. **客戶端 SDK 介面**
   - Python：`vacant_query(domain, weights)`, `vacant_call(agent_id, payload)`, `vacant_review(call_id, scores)`
   - 自動處理 envelope 簽章、Registry 查詢、結果聚合
   - 如何降級到「沒裝 vacant 也能做純 A2A 呼叫」

6. **OpenClaw plugin / Hermes toolset / Claude Code skill 整合方案**
   - 三個現存客戶端各給一段「如何接入 vacant」的設定範例
   - 客戶端**永遠不是 vacant**（強調這個區分）

7. **降級與失敗模式**
   - Registry 不可達時客戶端怎麼辦
   - 簽章驗證失敗 → 視為 untrusted call（標記但不拒絕）
   - 對非 Vacant 知曉的 A2A endpoint 呼叫時的兼容（gracefully fall back）

## 必答張力

- 客戶端與 vacant 的區分如何在規格上 enforce
