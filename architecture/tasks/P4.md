## 任務範圍

Registry 是 Vacant 唯一的有形中央元件。它「不思考、不仲裁、不參與互動」，但握有資料就是權力。你要設計：

1. **完整 Schema**
   - **Capability Card**：vacant_id, public_key, declared_capabilities, base_model, version, parent_id, attestations, stake, owner_org（與 P2 對齊）
   - **Event Log**（append-only）：call_event, review_event, peer_review_event, spawn_event, self_eval_event, sink_event
   - **Reputation Snapshot**：每維分數 + 信賴區間 + 樣本數 + 多元性指標（與 P3 對齊）
   - **Composition Link**：A 與 B 的雙邊產業鏈綁定
   - **Sink Record**：被沉沒的記錄（不刪除）

2. **使用 codex CLI 做研究**
   - 必查：MINJA 攻擊細節、eTAMP、append-only log 的密碼學承諾（hash chain、Merkle Tree、Certificate Transparency）、Sigstore / Rekor、Git 作為 timestamping anchor、Layer-2 commitment schemes
   - 輸出每個方案 (a) tamper-evidence 強度、(b) 寫入吞吐、(c) Vacant 適配性

3. **防竄改設計**（多層）
   - **層 1** 每筆 event 包含 hash(prev_event) → hash chain
   - **層 2** 週期性 Merkle root push 到 Git（Certificate Transparency 風格 timestamp）
   - **層 3** 多方 attestation：每個 event 至少 N 個獨立簽章才視為 finalized
   - **層 4** 異常偵測：rep 分數突跳的自動 freeze + 公示

4. **聯邦化演進**
   - **MVP**：單一 Registry process（SQLite + WAL）+ 公鑰索引 + Git commitment
   - **中期**：聯邦（多個機構各跑一個，互相同步 Merkle root + 衝突解決）
   - **長期**：完全分散化（CRDT / libp2p / on-chain anchor）
   - 每階段保持 vacant_id 跨遷移可攜（公鑰錨定不換）

5. **API 設計**
   - 寫入：`register_vacant`, `submit_event`, `submit_review`, `spawn`, `sink`
   - 讀取：`query_capability(domain, weights)`, `get_reputation(vacant_id, dim?)`, `get_event_log(vacant_id, range)`, `get_lineage(vacant_id)`
   - 查詢必須 paginated + 帶簽章 commitment（防止 Registry 偷改後返回）

6. **防 MINJA 流程**
   - vacant 寫入 review/spawn 必須由本機 vacant Runtime 直接簽章（不接受 third-party 代寫）
   - 所有寫入有 idempotency key + 重放保護
   - 若 vacant 被劫持（其私鑰外洩），freeze 機制 + 第三方 revocation list
