## 任務範圍

Vacant 訴求「無資格審核 + 失敗網路淘汰」，但這在開放網路中會立刻被 Sybil 與 whitewashing 拆毀。你要設計 **Vacant 識別層** 在「無中央仲裁」前提下抗這些攻擊。

具體要：

1. **使用 codex CLI 做深度研究**（你能直接呼叫 `codex` 命令；先 `codex --help` 看怎麼用）
   - 目標：盤點當前 LLM agent 身份/Sybil 抵抗的學術與產業方案
   - 必查：ETHOS arXiv:2412.17114、Bouchiha et al.、Friedman 2007、Douceur 2002
   - 補：SBT (Soulbound Tokens, Vitalik 2022)、SSI/DID (W3C)、企業 attestation patterns、Web of Trust、stake-based identity (Sybil-resistance via cost)
   - 輸出條列每個方案的 (a) 抗 Sybil 強度、(b) 對 Vacant 的可移植性、(c) 工程成本

2. **設計 Vacant 識別層**（多層防禦）
   - **L0 加密身份**：Ed25519 keypair → vacant_id（必選）
   - **L1 來源 attestation**：開發者/組織 sign 上線時的 Capability Card（必選）
   - **L2 Stake / Bond**：可選，誰想要更高 trust weight 才需要（與 reputation 進入算式相關，要與 P3 對齊）
   - **L3 Web of Trust**：vacant 之間的雙邊「我認得你不是 sock-puppet」簽章
   - 每層獨立，攻擊者要全突破才能 game
   - 要明確「whitewashing 的 cost 函數」公式

3. **冷啟動 reputation 從哪來（明確算式）**
   - 新 vacant 上線時的 prior（不是 0，也不是滿分）
   - 開發者「自我宣告」的歷史如何進入（低權重）
   - L1 attestation 等級對 prior 的影響

4. **聯邦化路徑**（Q1）
   - MVP：單一 Registry（公鑰錨定，所以 Registry 換不掉 vacant id）
   - 中期：聯邦 Registry 互相同步 Merkle root
   - 長期：完全分散化（IPFS / libp2p / SBT 鏈上）
   - 三階段的 trade-off 列表

5. **正面回答 BRIEFING §11 第 1 與第 2 個張力**
   - 「網路自然淘汰」如何不被 whitewashing 繞過？
   - 第一個 vacant 上線時 reputation 從哪來（沒有歷史）？
